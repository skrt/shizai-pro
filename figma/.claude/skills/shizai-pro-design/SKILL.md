# shizai pro デザインナレッジ

shizai proのプロダクトデザインに関する統合ナレッジ。画面構成、UIパターン、インタラクション定義を体系化したもの。

---

## 1. プロダクト概要

### shizai proとは

**shizaiの資材流通事業と一体運用される発注・資材管理SaaS**。shizaiに発注する資材の管理をメインの用途とし、発注クラウド管理、メッセージ管理、WMS（LOGILESS）連携、原価計算機能を提供する。

### ターゲットユーザー

- **デマンド（発注者側）**: shizaiの流通事業を利用するブランド企業の資材担当者・SCM担当者（SOEJU、COLORIS、HINEMOS、土屋鞄、tential、newnなど）
- **オペレーター（shizai側）**: shizaiの社内オペレーター（一部管理機能はオペレーター専用）

### 現在のビジネスコンテキスト（最新）

- **流通事業との一体運用が軸**: shizaiに発注する資材は全てshizai pro経由で管理し、発注データの一括管理でチャーンリスクを低減
- **商談の窓口拡大**: 資材の商談にshizai proも同時に持っていくことで接点を増やす戦略
- **機能開発の方向転換**: WMS連携前提の商品管理 → **資材管理に必要な機能**（在庫管理、自動発注、lubeとの連携）の拡充にシフト
- **lube連携**: shizai自社システム「lube」との連携機能の拡充が求められている
- 3プラン構成：Starter（無料〜）、Standard（5万/月）、Professional（10万/月）

### 技術スタック

- **リプレイス進行中**: Go → Rails への移行を実行中
- フロントエンド: Rails + TailwindCSS（独自コンポーネント）
- WMS連携: LOGILESS（API連携）
- 自社システム連携: lube（連携機能拡充中）
- 通知: メール + Slack連携

### 今後の機能拡充方向

- **在庫管理機能**: 資材の在庫状況をリアルタイムに把握
- **自動発注機能**: 在庫閾値に基づく発注サジェスト・自動化
- **lube連携強化**: shizai流通事業側のデータとのシームレスな連携

## 3. UIトーン・ビジュアルガイドライン

> **注意**: このセクションの「実測値」は現行 Go 版の UI から取得したもの。新 Rails 版ではセマンティックトークン体系（`/figma-create-component` 参照）に従うため、具体的な Hex 値やピクセル値は参考程度に留めること。デザインの**方針**と**パターン**は引き続き有効。

### デザインの方針

- **業務ツールとしての信頼感**: 派手さより正確性と視認性を重視
- **情報密度のバランス**: テーブルビューが多いため、適切な余白とグルーピングで可読性を確保
- **一貫性**: 全画面でCRUD操作パターン（一覧→追加→詳細→編集→削除）を統一
- **白基調のクリーンなUI**: 背景は白、ボーダーとテキストでコンテンツを構造化

### ブランド・カラーパレット（実測値 — Go版参考）

- **プライマリ/ブランドカラー**: シアン/ターコイズ系 — CTAボタン、アクティブなナビタブ、リンク色に使用
- **テキスト（primary）**: Gray-800〜900（ほぼブラック）
- **テキスト（secondary）**: Gray-500〜600（ラベル、サブ情報）
- **テキスト（placeholder）**: Gray-400
- **リンク色**: シアン系
- **エラー/必須**: Red（赤アスタリスク `*` で必須を表示）
- **背景**: White（メイン）、入力フォームはWhite + Grayボーダー
- **ボーダー**: Gray-200〜300（テーブル行区切り、フォーム枠）

### ボタンスタイル（実測値 — Go版参考）

| 種類 | スタイル | 用途 |
|------|---------|------|
| プライマリCTA | シアン背景 + 白文字、角丸 | 新規発注、新規アイテム追加、確認画面へ進む、認証、再連携 |
| セカンダリ | 白背景 + Grayボーダー + Black文字 | 発注取り消し、複製、下書き保存 |
| テキストボタン | ボーダー付き白背景 | アイテムをまとめて選択、CSVで入力、行追加 |
| 詳細ボタン | シアン背景 + 白文字、小さめ | テーブル行の「詳細」遷移 |
| Disabledボタン | シアンの薄い半透明 | 確認画面へ進む（未入力時） |

### ステータスバッジカラー（実測値 — Go版参考）

| ステータス | 表示スタイル | 備考 |
|-----------|------------|------|
| 下書き | テキストのみ（バッジなし） | フィルターボタン |
| 承認待ち | オレンジ/Amber バッジ + 件数 | 件数表示あり |
| 送信待ち | 件数付きテキスト | 件数表示あり |
| 納品待ち | オレンジ背景 + 白文字バッジ + 件数 | テーブル内でも目立つ |
| 会話中 | 件数付きテキスト | フィルターボタンで件数表示 |
| 納品済み | グレー背景バッジ | 完了状態、テーブル内 |
| 差し戻し | テキストのみ | フィルターボタン |
| 取り消し | テキストのみ | フィルターボタン |

### ナビゲーション構造

- **ヘッダー**: 左に「shizai」ロゴ、右に「会社名 + ユーザー名」「ヘルプ」
- **グローバルナビ**: ヘッダー下にタブ形式（発注 | 納品 | 原価計算 | アイテム | 仕入先 | 納品先 | メンバー | 設定）
  - アクティブタブ: シアン色テキスト + 下線
  - 非アクティブタブ: Grayテキスト
- **戻るボタン**: 詳細画面に「←」アイコンボタン

### レイアウト

- **コンテンツ幅**: 左右にパディングあり、最大幅制限あり（左寄せ）
- **テーブル**: ボーダーレス、行間にライトグレーの区切り線
- **フォーム**: ラベルは上配置、入力フォームは薄いGrayボーダーの角丸ボックス
- **必須表示**: ラベル横に赤い `*`
- **2カラムレイアウト**: 発注書詳細画面では左に発注内容、右に会話パネル

### タイポグラフィ

- **ページタイトル**: 太字、大きめ
- **セクション見出し**: 太字
- **ラベル**: normal weight、Gray色
- **テーブルヘッダー**: 小さめ、Gray色
- **テーブル数値（単価・金額・数量）**: 右揃え、カンマ区切り表示
- **日付**: yyyy/mm/dd(曜日) 形式（例: 2025/12/01(月)）

## 4. 共通UIパターン

### 4.1 マスタCRUDパターン（一覧 → 追加 → 詳細 → 編集 → 削除）

仕入先・納品先・アイテムで共通のパターン:

**一覧**
- テーブル表示、ページング（デフォルト10件、切替可能・状態保存）
- 情報がないセルは「なし」または「-」表示
- 各行に詳細遷移ボタン

**追加**
- フォーム画面、必須/任意の明示
- WMS連携チェックボックス（連携時のみ表示、デフォルトチェック済み）
- 作成完了 → 一覧に戻る

**詳細**
- 登録情報の表示 + 関連発注一覧テーブル
- 編集・削除ボタン

**編集**
- 追加フォームと同構成、ボタンラベルが「更新」に変わる
- WMS連携状態は変更不可
- WMS連携時の注意メッセージ表示

**削除**
- 確認ダイアログ表示
- WMS連携時: 削除ボタンDisable + Tooltip説明（アイテム）またはメッセージ表示（仕入先・納品先）

### 4.2 検索パターン

| パターン | 使用箇所 | 動作 |
|---------|---------|------|
| ComboBox検索 | 仕入先選択、アイテム選択 | テキスト入力可、100件表示、件数表示、マスタ追加ボタン付き |
| Select選択 | 納品先選択、都道府県 | テキスト入力不可、全件表示 |
| テキスト検索 | 発注書番号、アイテムフィルター | 入力ごとに検索実行 |
| カレンダー選択 | 発注日、希望納品日 | DatePicker、期間指定（開始日-終了日） |
| ステータスボタン | 発注書一覧 | ボタン切替、件数バッジ（承認待ち/送信待ち/納品待ち/会話中） |

### 4.3 フォーム共通ルール

- 入力前にフォーカス状態を明示（border-focus + ring-focus）
- 必須項目は明示（ラベル横にアスタリスク等）
- バリデーションエラーはフィールド直下に表示
- 汎用エラーはトースト表示
- 削除ボタンがある入力フォームは1件の場合Disable
- モーダルを閉じた場合、入力内容は初期化される

### 4.4 テーブル共通ルール

- 数値は右揃え、カンマ区切り（10,000形式）
- 単価は小数点第2位まで（アイテムマスタは第6位まで）
- 金額は納品数 × 最終単価、小数点以下切り捨て、税別
- ページング: デフォルト10件、表示件数切替の状態を保存
- チェックボックス: 一括操作用、全選択/個別選択、ページ跨ぎ不可

### 4.5 フィードバックパターン

| 種類 | 用途 | 実装 |
|------|------|------|
| トースト | 操作完了/失敗の通知 | 画面上部に一時表示 |
| Skeleton | ローディング中 | テーブル・コンテンツ部分 |
| アラート | 画面離脱時の未保存警告 | ブラウザのアラート |
| Tooltip | 操作不可の理由説明 | ボタンDisable時にホバーで表示 |
| エラーメッセージ | バリデーションエラー | フィールド直下に赤字表示 |

## 5. 機能別デザイン仕様

### 5.1 発注書一覧（メイン画面）

**フィルター構成**:
- 発注書番号（テキスト入力、yyyymmdd-xxx形式）
- アイテム名/件名（テキスト入力、「スペースで複数入力できます」とプレースホルダ）
- 仕入先（ComboBox検索、「すべて」プレースホルダ）
- 納品先（Select選択、「すべて」デフォルト）
- 発注日（カレンダー期間指定、「全期間」プレースホルダ）
- 希望納品日（カレンダー期間指定、「全期間」プレースホルダ）

**ステータスフィルター**:
- 横並びボタン形式: すべて | 下書き | 承認待ち(件数) | 送信待ち(件数) | 納品待ち(件数) | 会話中(件数) | 納品済み | 差し戻し | 取り消し
- アクティブタブ: シアン枠のpillボタン（「すべて」）
- 件数バッジ: 承認待ち=赤文字、送信待ち=テキスト、納品待ち=赤文字、会話中=シアン文字

**フィルターの挙動**:
- AND検索のみ（OR不可）
- 画面更新/遷移後もフィルター状態を維持
- ページング位置も維持
- リセットボタンで初期状態に復帰

**テーブルカラム構成**:
- チェックボックス | アイコン | 発注書番号 | 入荷予定コード(リンク) | 件名 | ステータス(バッジ) | 発注日 | 仕入先(リンク) | 納品先(リンク) | 合計金額(右揃え) | 会話(メールアイコン) | 詳細(ボタン)

**一括操作**:
- チェックボックスによる複数選択
- ステータス一括変更（納品待ち <-> 納品済み）
- 会話ステータス一括変更（会話中 <-> 会話終了）
- 承認待ち/送信待ち/差し戻しはチェックボックスDisable
- ページ跨ぎの選択は不可

### 5.2 発注書作成/編集

**画面構成**:
- 上部: 戻るボタン + 「発注取り消し」「複製」ボタン + メール通知アイコン
- 左カラム: 発注書の詳細 + 発注内容フォーム
- 右カラム（サイドパネル）: 「仕入先にメッセージを送る」会話パネル

**ヘッダー情報（詳細画面）**:
- 発注書番号、入荷予定コード（リンク）、作成者
- ステータス（ドロップダウンで変更可能）
- 承認フロー表示（申請→承認→送信の3ステップ、完了マーク付き）
- 社内メモ（テキストエリア）
- データ添付ボタン

**ヘッダー情報（新規作成画面）**:
- 仕入先（ComboBox、「検索」プレースホルダ）、納品先（ComboBox、「検索」プレースホルダ）
- 件名（テキスト入力）、発注日（DatePicker、当日がデフォルト）

**明細行**:
- カラム構成: チェックボックス | 種類（アイテム/費用 ラジオ）| 品名 | 数量 | 単価 | 税区分 | 希望納品日 | 支払期限 | 詳細展開 | 削除
- アイテム行: ComboBox検索（名前/商品コードで検索可、選択後「アイテム名 / ID」表示、クリアボタン）
- 費用行: 種類が費用に切り替わり、品名は「費用名」のフリーテキストInput
- 単価: 半角数字+「.」、小数点第6位まで、0円可、マスタから自動入力
- 数量: 半角数字のみ、自然数のみ、0不可
- 税区分: Select（10%/軽減8%/対象外）、デフォルト10%
- 希望納品日: DatePicker
- 支払期限: DatePicker（希望納品日の翌月末日を自動入力）
- 詳細展開: メモ（256文字）/ ロット番号（15桁）/ 出荷期限日

**単価自動入力**:
- アイテム選択時、マスタの初期値単価を自動入力
- 既入力値は上書きされる
- アイテムクリア後も単価は残る

**単価表ポップアップ**:
- 数量/単価フォームにフォーカスで表示
- 行選択で数量・単価を上書き

**チェックボックスによる同時編集**:
- 複数行選択 → 1箇所の変更が全選択行に反映
- アイテム/費用の種類も連動して切替

**明細行追加操作**（3つのボタンが横並び）:
- 「アイテムをまとめて選択」
- 「CSVで入力」（ドロップダウン付き）
- 「行追加」

**フッター部分**:
- 補足テキストエリア（リサイズ可能）
- 小計・消費税（税率別内訳）・合計金額（右寄せ）
- 「下書き保存」（セカンダリボタン）+ 「確認画面へ進む」（プライマリCTA、全幅）

**一括アイテム選択モーダル**:
- タイトル: 「発注アイテムを選択」
- 検索フォーム（テキスト入力 + クリアボタン）
- 「検索結果：N件」の件数表示
- 「{入力テキスト}を追加」シアンリンク（最上部）
- テーブル: チェックボックス | アイテム名（下にID/商品コード） | カラー | サイズ
- フッター: 「N件選択中」（左）+「選択完了」ボタン（右、シアンCTA）
- モーダルを閉じてもチェック状態を維持
- スクロール可能なリスト

**CSVアップロード**: 明細をCSVで一括入力可能

### 5.3 ステータスフロー

```
下書き → 承認待ち → [承認] → 送信待ち → [送信] → 納品待ち <-> 納品済み
                   → [差し戻し] → 差し戻し → [再申請] → 承認待ち
```

**明細行ステータス**（WMS連携時のみ表示）:
- 納品待ち → 一部納品済み → 納品済み
- 全明細行が納品済み/一部納品済み → 発注書ステータスが自動で納品済みに

**管理者は申請不要で直接送信可能**

### 5.7 WMS連携（LOGILESS）

- 1日1回朝5時に自動マスタ同期

- 連携済みアイテムは名称・カラー・サイズ・単価の変更不可

## 7. ユーザーフロー・インタラクション定義

### 発注フロー（編集者）

```
発注書一覧 → 新規作成 → 明細入力 → 下書き保存
→ 承認申請 → [承認者が承認] → 送信待ち
→ 確認画面（送信先メール確認 + 発注書プレビュー + データ添付）
→ 発注書送信 → 納品待ち
→ [WMS同期で納品確認] → 納品済み
```

### 発注フロー（管理者）

```
発注書一覧 → 新規作成 → 明細入力
→ 確認画面（送信先メール確認 + 発注書プレビュー + データ添付）
→ 発注書送信（承認不要）→ 納品待ち → 納品済み
```

### 会話フロー

```
発注書詳細 → 会話エリア → メッセージ入力 → 送信
→ 仕入先にメール送信 → 仕入先が返信 → 会話履歴に反映
→ 「会話中」ステータスON → 手動でOFF
```

### アカウント登録フロー

```
招待メール受信 → リンククリック（6時間有効）
→ メールアドレス確認 → 氏名登録 → パスワード設定 → 完了
```

### WMS連携フロー

```
設定 → マーチャントID入力 → LOGILESS認証
→ マスタデータ取得開始 → 完了通知（Slack/メール）
→ 以降、毎朝5時自動同期 + 手動再連携可能
```

## 8. デザイン判断の原則

新しい機能や画面をデザインする際に従うべき原則:

1. **資材管理が中心**: アパレル単独のWMS前提ではなく、shizai流通事業と一体の「資材管理」がメインの用途。在庫管理・自動発注・lube連携を見据えた設計を行う
2. **既存パターンの踏襲**: 一覧→詳細→編集のCRUDパターン、ComboBox/Select/DatePickerの使い分けは既存を必ず踏襲する
3. **Railsリプレイスを意識**: Go→Railsへの移行中。新規画面はRails + TailwindCSSで実装されることを前提にデザインする。ERBパーシャルやViewComponentへの分解しやすさを考慮する
4. **WMS連携の考慮**: 全マスタ画面でWMS連携時の制約（編集不可、削除不可）を設計に含める
5. **権限の考慮**: 管理者/編集者で見える画面・操作できるアクションが異なることを常に意識する
6. **通知への影響**: 画面上のアクション（送信、更新、取消）が誰にどの通知を発生させるかを把握する
7. **お金の情報の取り扱い**: 納品先にはお金の情報を見せない原則を守る
8. **CSVの非同期パターン**: 大量データのCSV出力は非同期パターン（作成→ダウンロード）を採用する
9. **フィルターの状態保持**: フィルター・ページングの状態は画面遷移後も維持する
10. **バリデーションの一貫性**: 文字数制限、入力形式、重複チェックのルールは既存仕様に合わせる
11. **lube連携の考慮**: 今後lube（自社流通管理システム）とのデータ連携が増えるため、データの入出力ポイントを意識した設計を行う
